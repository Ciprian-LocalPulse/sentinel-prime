# Contributing Policy

Thank you for your interest in Sentinel Prime. This repository is maintained as a controlled Public Edition and premium cybersecurity product evaluation surface.

## Contribution Model

External code contributions are not accepted unless explicitly approved by the repository owner in advance.

This restriction protects the Public Edition and prevents accidental disclosure of private enterprise implementation details, commercial materials, dashboard logic, detection concepts, telemetry assumptions, customer environments, or security-sensitive information.

## Contributions That May Be Considered

The maintainer may selectively consider non-sensitive contributions such as:

- typo fixes or documentation clarity improvements
- governance file improvements
- accessibility notes for public documentation
- issue reports about broken links or inaccurate public claims
- safe suggestions for public repository hygiene

Please open an issue before preparing any pull request. Unapproved pull requests may be closed without review.

## Contributions That Will Not Be Accepted

Do not submit, request, or describe:

- dashboard logic, production SOC workflows, detection rules, or triage logic
- threat-intelligence feeds, telemetry pipelines, enrichment logic, or event schemas
- incident-response playbooks, analyst runbooks, or internal operating procedures
- authentication logic, access-control rules, tokens, credentials, environment variables, or secrets
- customer data, incident records, client environments, internal network references, or infrastructure diagrams
- exploit instructions, evasion guidance, persistence techniques, offensive workflows, or unauthorized testing methods
- private prompts, restricted research, commercial implementation details, pricing, or buyer-specific requirements
- changes that convert the repository from premium cybersecurity product positioning into an operational dual-use toolkit

## Public Edition Boundary

The Public Edition communicates safe product direction, premium cybersecurity interface quality, architecture-level thinking, governance maturity, and commercialization readiness. It is not the Private Enterprise Edition and does not accept contributions that attempt to recreate private implementation details.

## Pull Request Expectations

If the maintainer explicitly invites a contribution:

1. Keep the change narrow and documentation-focused unless otherwise approved.
2. Do not modify design, visuals, dashboard logic, assets, README product positioning, or private implementation details unless specifically requested by the maintainer.
3. Avoid adding dependencies, scripts, data collection, telemetry, detection logic, integration behavior, or executable behavior.
4. Keep language calm, enterprise-safe, CISO-friendly, procurement-safe, and legally careful.
5. Confirm that no secrets, credentials, customer references, incident details, private dashboard logic, or private methods are included.

## Issues

Public issues should be limited to non-sensitive repository concerns. Do not open public issues containing vulnerability details, detection rules, telemetry logic, customer information, private architecture, implementation-sensitive details, or commercial negotiation material. Use SECURITY.md for security reporting.

## License and Ownership

All repository materials remain proprietary unless explicitly stated otherwise in LICENSE.md or a separate written agreement. Contributions, if accepted, may be incorporated into proprietary materials without creating ownership, licensing, or commercialization rights for the contributor.

## Maintainer Discretion

The maintainer may close issues or pull requests that are out of scope, unsafe, duplicative, commercially sensitive, legally sensitive, or inconsistent with the premium cybersecurity product positioning of this repository.
